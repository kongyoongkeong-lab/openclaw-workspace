## WSL2 ‚Üí Chrome Remote Debugging Port 9222 ‚Äî Investigation

### Problem
WSL2 (NAT network) cannot reach Windows host's Chrome at 127.0.0.1:9222. All attempts to connect failed.

### Solutions Found (from GitHub / web research)
1. **WSL2 `networkingMode=mirrored`** ‚Äî Edit `%USERPROFILE%\.wslconfig`, add `[wsl2]\nnetworkingMode=mirrored`, then `wsl --shutdown`. WSL2 and Windows share the same network stack. Requires Windows 11 22H2+.

2. **`netsh` portproxy** ‚Äî From Windows admin: `netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=9222 connectaddress=127.0.0.1 connectport=9222`. Then connect from WSL2 via Windows host IP (found via `ip route | grep default | awk '{print $3}'`).

3. **Windows Firewall + portproxy combo** ‚Äî First add firewall rule for inbound 9222, then add portproxy.

4. **Chrome launch directly** ‚Äî `google-chrome --remote-debugging-port=9222` from within WSL2 (if Chrome Linux is installed in WSL2).

5. **SSH reverse tunnel** ‚Äî `ssh -R 9222:localhost:9222 wsl-host` (over-engineered, not recommended).

6. **Use `socat` within WSL2** ‚Äî `socat TCP-LISTEN:9222,fork TCP:$(ip route | grep default | awk '{print $3}'):9222` (only works if Windows port is reachable, which it wasn't).

### Verified State (as of 19:05 MYT)
- Chrome running on Windows with `jasonykkaiykk@gmail.com` profile
- Firewall rule added: `Chrome Debug` inbound TCP 9222
- Port still unreachable from WSL2 ‚Äî NAT isolation confirmed
- OpenClaw headless browser (localhost:18800) works reliably for public websites

### Best Recommendation
**Primary:** `networkingMode=mirrored` in `.wslconfig` + `wsl --shutdown` ‚Äî most robust, no firewall/portproxy needed.
**Fallback:** `netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=9222 connectaddress=127.0.0.1 connectport=9222` (after enabling listener on 0.0.0.0).
**Current workaround:** OpenClaw headless browser works for all public sites; login-required sites need manual credential entry once.

## WSL2 ‚Üí Chrome Remote Debugging Port 9222 ‚Äî Investigation

### Problem
WSL2 (NAT network) cannot reach Windows host's Chrome at 127.0.0.1:9222. All attempts to connect failed.

### Solutions Found (from GitHub / web research)
1. **WSL2 `networkingMode=mirrored`** ‚Äî Edit `%USERPROFILE%\.wslconfig`, add `[wsl2]\nnetworkingMode=mirrored`, then `wsl --shutdown`. WSL2 and Windows share the same network stack. Requires Windows 11 22H2+.

2. **`netsh` portproxy** ‚Äî From Windows admin: `netsh interface portproxy add v4tov4 listenaddress=0.0.0.0 listenport=9222 connectaddress=127.0.0.1 connectport=9222`. Then connect from WSL2 via Windows host IP (found via `ip route | grep default | awk '{print $3}'`).

3. **Windows Firewall + portproxy combo** ‚Äî First add firewall rule for inbound 9222, then add portproxy.

4. **Chrome launch directly** ‚Äî `google-chrome --remote-debugging-port=9222` from within WSL2 (if Chrome Linux is installed in WSL2).

5. **SSH reverse tunnel** ‚Äî `ssh -R 9222:localhost:9222 wsl-host` (over-engineered, not recommended).

6. **Use `socat` within WSL2** ‚Äî `socat TCP-LISTEN:9222,fork TCP:$(ip route | grep default | awk '{print $3}'):9222` (only works if Windows port is reachable, which it wasn't).
